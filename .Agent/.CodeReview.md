# Code Review Guidelines

> **Purpose:** Define mandatory review checklist, severity criteria, and blocking rules  
> **Authority:** This document is a binding contract for all code changes

---

## Review Checklist

### TypeScript & Type Safety
- [ ] No `any` types used
- [ ] No forced casting (`as Type`)
- [ ] All functions have explicit return types
- [ ] All parameters are typed
- [ ] Interfaces preferred over types for objects
- [ ] Generics used appropriately

### React Components
- [ ] Server Components used by default
- [ ] `'use client'` only when necessary
- [ ] Props interfaces defined
- [ ] Component props destructured
- [ ] `key` prop provided for lists
- [ ] `useMemo`/`useCallback` used for expensive operations
- [ ] `memo` used for pure components

### Zod Validation
- [ ] All API inputs validated
- [ ] Error messages in Portuguese (pt-BR)
- [ ] Types inferred from schemas
- [ ] Validation at API boundary

### Database (Prisma)
- [ ] Soft delete used instead of hard delete
- [ ] Transactions for multi-step operations
- [ ] Proper error handling
- [ ] Include statements for relationships when needed

### API Routes
- [ ] Error handling with try/catch
- [ ] Proper HTTP status codes
- [ ] Consistent response format
- [ ] Rate limiting considered
- [ ] Authentication checked

### Styling (Tailwind)
- [ ] `cn()` utility used for conditional classes
- [ ] No arbitrary values without justification
- [ ] Responsive design considered
- [ ] Dark mode support (if applicable)

### Testing
- [ ] Unit tests for utilities
- [ ] Component tests for UI
- [ ] Mock external dependencies
- [ ] Test edge cases

### Performance
- [ ] No unnecessary re-renders
- [ ] Lazy loading for heavy components
- [ ] Images optimized
- [ ] Bundle size considered

### Security
- [ ] No secrets in code
- [ ] Input sanitization
- [ ] SQL injection prevention (Prisma handles this)
- [ ] XSS prevention

### Documentation
- [ ] JSDoc for public APIs
- [ ] README updated if needed
- [ ] Complex logic explained

---

## Severity Levels

| Level | Description | Action |
|-------|-------------|--------|
| **CRITICAL** | Security vulnerability, data loss risk | BLOCK merge |
| **HIGH** | Type safety violation, broken functionality | BLOCK merge |
| **MEDIUM** | Performance issue, missing tests | Must fix |
| **LOW** | Style inconsistency, minor optimization | Suggested |
| **INFO** | Suggestions, questions | Optional |

---

## Blocking Rules

### Automatic Block (CRITICAL/HIGH)

A PR MUST be blocked if it contains:

- [ ] `any` type usage
- [ ] `console.log` in production code
- [ ] Hardcoded secrets or credentials
- [ ] SQL injection vulnerabilities
- [ ] XSS vulnerabilities
- [ ] Broken TypeScript compilation
- [ ] Failed tests
- [ ] Missing error handling in API routes
- [ ] Direct state mutation
- [ ] Missing validation on API inputs

### Conditional Block (MEDIUM)

A PR SHOULD be blocked if it contains:

- [ ] Missing tests for new features
- [ ] Performance regressions
- [ ] Accessibility issues
- [ ] Missing JSDoc for public APIs

## Review Process

### 1. Pre-Review (Automated)
```bash
# Run before requesting review
npm run lint        # ESLint checks
npm run type-check  # TypeScript compilation
npm run test        # Unit tests
npm run build       # Production build
```

### 2. Review Assignment
- All PRs require at least 1 reviewer
- Complex changes require 2 reviewers
- Security-related changes require senior reviewer

### 3. Review Cycle
```
Author → Reviewer → Comments → Author → Changes → Reviewer → Approve → Merge
```

### 4. Approval Criteria
- [ ] All CRITICAL/HIGH issues resolved
- [ ] All automated checks passing
- [ ] At least 1 approval from code owner
- [ ] No unresolved conversations

## AGENT_REVIEWER Responsibilities

When acting as AGENT_REVIEWER:

1. **Verify compliance** with `.CodeConventions.md`
2. **Check against** `.CodeReview.md` checklist
3. **Report findings** using standard format
4. **Never approve** if blocking rules violated
5. **Suggest improvements** even for approved code

### Review Scope

**Within Scope:**
- Code quality
- Type safety
- Performance
- Security
- Test coverage
- Documentation

**Out of Scope:**
- Business logic validation (unless obviously wrong)
- UI/UX design decisions
- Architecture changes (flag for discussion)

## Avisos-specific Review Criteria

### Tipo Field Validation
- [ ] Aviso tipo is properly typed as `'Aviso' | 'Publi'`
- [ ] Default value is 'Publi' when not specified
- [ ] Database enum matches TypeScript type

### Dual View System
- [ ] URL param `?tipo=aviso` shows Avisos view (notification style)
- [ ] URL param `?tipo=publi` shows Publi view (advertising style)
- [ ] View mode is determined from searchParams
- [ ] Suspense boundary wraps component using useSearchParams

### Carousel Card Layouts
- [ ] Publi cards display image on top, title at bottom
- [ ] Aviso cards display title on top, description in place of image
- [ ] URL hyperlink is visible on Aviso cards

### View-specific Features
**Avisos View:**
- [ ] Shows notification/message style cards
- [ ] Has unread indicator (blue dot)
- [ ] Shows message icon
- [ ] Title reads "Mural de Avisos"

**Publi View:**
- [ ] Shows advertising card style
- [ ] Carousel displays at top
- [ ] Title reads "Espaço de Publicidades"

### Auto-set Tipo
- [ ] Creating in Avisos view auto-sets tipo to 'Aviso'
- [ ] Creating in Publi view auto-sets tipo to 'Publi'
